SSE 是做什么的 现实中 SSE 的应用领域

是一种**服务器向浏览器单向推送数据**的技术，基于 HTTP 协议，使用简单的事件流格式。

    • 流式响应更自然

      • SSE天然支持流式输出，无需复杂的流控制

      • Websocket需要手动实现流处理

    • 自动重连

      • SSE浏览器自动重试

      • Websocket需客户端实现重连逻辑

  1. Promise 有哪些常用方法？

浏览器中的 JavaScript 受限于沙盒环境，只能操作客户端（如页面交互、简单网络请求），无法直接访问文件、系统资源或底层网络（如 TCP、数据库）。而 Node.js 是服务端 JavaScript 运行环境，它突破浏览器限制，提供文件读写、数据库操作、高并发网络通信等服务端核心能力；同时凭借“单线程+事件驱动”模型高效处理 I/O 密集型场景（如 API 接口、实时聊天），并通过统一前后端技术栈（JS 全栈）降低开发成本。简言之：浏览器 JS 管“用户界面”，Node.js 管“服务逻辑”，两者互补，Node.js 让 JS 从客户端脚本升级为全栈开发工具。

Node.js怎么实现接收文件，写入服务器的操作

在 Node.js 中接收并写入文件，常用 <font style="color:#1DC0C9;">Express + multer</font>方案。核心步骤：

Vue3响应式

过 <font style="color:#1DC0C9;">ref</font>（基本类型/对象）和 <font style="color:#1DC0C9;">reactive</font>（对象）两个 API 实现数据响应。<font style="color:#1DC0C9;">Proxy</font>能拦截对象所有操作（包括属性增删、数组索引修改），配合依赖收集（<font style="color:#1DC0C9;">Track</font>）和触发更新（<font style="color:#1DC0C9;">Trigger</font>）机制，实现更高效、全面的响应式系统（通过套着对象的代理对象来拦截对属性的操作）

ref和reactive的区别

ref：用于基本类型（如 <font style="color:#1DC0C9;">string</font>、<font style="color:#1DC0C9;">number</font>）或对象，通过 <font style="color:#1DC0C9;">.value</font>访问/修改值（对象本质是 <font style="color:#1DC0C9;">Proxy</font>包装的 <font style="color:#1DC0C9;">RefImpl</font>对象）。

reactive：仅用于对象（引用类型），直接访问属性（无需 <font style="color:#1DC0C9;">.value</font>），底层通过 <font style="color:#1DC0C9;">Proxy</font>拦截所有操作。

Proxy 包装对象是什么

Proxy 包装对象是 ES6 提供的“对象代理器”，通过 <font style="color:#1DC0C9;">new Proxy(target, handler)</font>创建，表面与原对象（<font style="color:#1DC0C9;">target</font>）行为一致，但所有操作（如属性访问、修改、删除）会被 <font style="color:#1DC0C9;">handler</font>中定义的拦截方法（如 <font style="color:#1DC0C9;">get</font>、<font style="color:#1DC0C9;">set</font>）自定义处理。它像一个“透明包装层”，实际操作会转发到原对象，但行为由 <font style="color:#1DC0C9;">handler</font>控制。

什么是组合式API

组合式 API 是 Vue3 推出的基于函数的 API 设计模式，替代了 Vue2 的选项式 API（如 <font style="color:#1DC0C9;">data</font>、<font style="color:#1DC0C9;">methods</font>）。它通过“按逻辑组织代码”而非“按选项类型拆分”，允许开发者将相关状态、方法和生命周期封装为可复用的函数（组合函数），实现更灵活的逻辑复用、更清晰的代码结构和更强大的类型支持。

Vue的生命周期钩子，有什么用

Vue 的生命周期钩子是组件在创建、更新、销毁过程中自动触发的“事件回调”，开发者可在这些钩子中插入自定义逻辑（如数据请求、DOM 操作、资源释放）。它像组件的“生命日记”，记录了组件从生到死的完整过程，确保逻辑在正确时机执行。用于在组件渲染的各个阶段进行一些异步操作，确保组件行为符合预期

什么是keepalive

用于在切换视图时，缓存组件，避免重新渲染，从而提升性能，有actived和deactived两个钩子函数，可以用include和exlude属性来决定缓存那些组件

如何对vue代码进行测试，非人为测试

单元测试：单个函数，组件或逻辑模块，确保独立运行正确（Jest，Vitest）

集成测试：关注多个模块或组件之间的交互逻辑，常用工具与上述一致

端到端测试：E2E 测试验证完整业务流程（如登录、下单），模拟用户在浏览器中的真实操作。常用工具为 Cypress 或 Playwright（跨浏览器支持更好）。

浏览器缓存策略分为强制缓存（直接使用本地缓存，由 <font style="color:#1DC0C9;">Cache-Control</font>/<font style="color:#1DC0C9;">Expires</font>控制）和协商缓存（需服务器确认有效性，由 <font style="color:#1DC0C9;">ETag</font>/<font style="color:#1DC0C9;">Last-Modified</font>控制）。缓存位置包括磁盘缓存（Disk Cache）（硬盘存储，容量大、生命周期长）和内存缓存（Memory Cache）（内存存储，速度快、易丢失）。磁盘与内存缓存的区别体现在存储介质、速度、容量和生命周期上，浏览器根据资源类型、访问频率、生命周期等因素决定存储位置，以平衡速度与资源利用率。

详细解释：

强缓存：浏览器直接用本地缓存，不向服务器发送请求

触发调节，响应头中的 cache-control 来说明缓存有效期

协商缓存：缓存过期之后，浏览器携带缓存标识向服务器验证资源是否更新，没更新就用本地缓存。通过响应头中的 <font style="color:#1DC0C9;">ETag</font>/<font style="color:#1DC0C9;">If-None-Match</font>或 <font style="color:#1DC0C9;">Last-Modified</font>/<font style="color:#1DC0C9;">If-Modified-Since</font>验证。

  1. 追问磁盘缓存和内存缓存区别，为什么有的缓存在磁盘有的缓存在内存。

磁盘缓存：读取速度慢，生命周期长（可能跨标签页，跨浏览器重启，直到缓存策略过期或手动清理），容量大，适合大文件，低频访问但长期需要的资源

内存缓存：读取速度快，容量小，生命周期短（关闭标签页之后可能就释放，浏览器会主动清理），适合高频访问的小文件，比如当前文件的HTML，css，js

事件循环是 JavaScript 运行时（Runtime）处理异步操作的核心机制，其核心职责是协调同步任务与异步任务的执行顺序，确保代码在单线程环境下（浏览器/Node.js）。

调用栈，js执行同步代码的地方，每次调用一个函数都会把当前的执行上下文压入栈顶，执行完成后，其上下文又弹出。

任务队列，异步任务的一个等待去，任务队列分为宏任务队列和微任务队列，当同步任务执行完且异步操作完成的时候，回把对应的回调函数放入队列等待。任务队列又分为宏任务队列和微笑任务队列。微任务的执行优先级更高

事件循环就是不断的检查调用栈是否未空，然后按照优先级从队列中去除任务进行执行。

核心顺序：

  1. 先执行调用栈中的所有同步任务；
  2. 调用栈清空后，检查微任务队列，按顺序执行所有微任务；
  3. 微任务队列清空后，检查宏任务队列，取出一个宏任务执行（浏览器通常取最早入队的宏任务，Node.js 按阶段处理）；
  4. 重复步骤 2~3（宏任务执行后可能产生新的微任务，需再次清空微任务队列）

首先，js中的任务分为了同步任务和异步任务，异步任务又分为了宏任务和微任务。这些任务执行的优先级为同步任务，微任务，宏任务。同时该机制提供了两个队列，一个宏任务队列，一个微任务队列，当js代码执行时，首先将同步代码执行完毕，一些异步操作执行完毕，微任务操作的回调函数放入微任务队列中，宏任务的回调放入宏任务队列中，当同步代码执行完毕后，先检查微任务队列，将微任务队列中的所有任务按顺序取出执行后，再检查宏任务队列，取出一个宏任务回调执行完毕后，可能又会产生微任务，所以需要再次执行微任务，一直不断的重复第二，三个步骤，直到任务队列全都清空就是一次事件循环

  1. react的fiber架构有啥用?为什么可以打断任务执行

核心：“可中断的增量更新”：

iber 的本质是将渲染工作拆分为多个“可独立调度的小单元”，通过“增量更新”和“优先级调度”实现更流畅的用户体验。其核心价值体现在三点：

**解决长任务阻塞，提升页面响应性**

旧版 Stack Reconciler 采用**同步递归**的方式遍历组件树，一旦开始渲染（如更新一个长列表），就会一直占用主线程，直到所有组件都处理完毕。这会导致：
  • 用户输入（如点击、滚动）被阻塞，页面“卡顿”；
  • 动画掉帧，体验不流畅。

Fiber 则将渲染拆分为多个 Fiber 节点（每个节点对应一个组件的更新任务），每次只处理一小部分 Fiber（比如一个组件或一个列表项），然后主动中断，让主线程处理高优先级任务（如用户事件）。等高优先级任务完成，再从上次中断的地方继续。这种“增量更新”彻底解决了长任务阻塞的问题。

2. **支持优先级调度，保证关键任务优先**

Fiber 为不同类型的更新分配**优先级**（如用户输入是“高优先级”，数据加载是“低优先级”）。当高优先级任务到来时，Fiber 会：
  • 暂停当前的低优先级更新；
  • 先处理高优先级任务（如用户点击按钮的回调）；
  • 高优先级任务完成后，再恢复低优先级更新。

这使得用户交互始终保持响应，不会出现“点击按钮后要等几秒才反应”的情况。

3. **优化性能，减少不必要的计算**

Fiber 通过**双缓存 Fiber 树**（`current`和 `workInProgress`）复用状态：
  • <font style="color:#01B2BC;">current</font>树：当前渲染到页面的 Fiber 树；
  • <font style="color:#01B2BC;">workInProgress</font>树：正在计算的下一帧 Fiber 树。

每次更新时，React 会基于 <font style="color:#01B2BC;">current</font>树生成 <font style="color:#01B2BC;">workInProgress</font>树，计算差异（diff），然后将 <font style="color:#01B2BC;">workInProgress</font>树提交到 DOM（成为新的 <font style="color:#01B2BC;">current</font>树）。这种机制避免了重复计算，提升了 diff 效率。
